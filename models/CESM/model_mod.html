Overview

The Lanai release of DART works within the CESM framework. The release includes a setup script that will build the required DART executables, build the CESM case, and copy the required initial files and executables to the correct locations. The CESM run script is modified during the build process so it first runs CESM, then it runs the DART executable 'filter'.

The instructions on this page describe how to set up a coupled CESM run when you want to assimilate observations into more than one active component (e.g. assimilate atmospheric obs into the atmosphere model, ocean obs into the ocean model, and land obs into the land model). If you are interested in assimilating observations into only a single component there are more appropriate setup scripts in each of the model-specific pages that set up a CESM run for that component.

See these component specific model_mod documentation for information on the namelists and requirements:

  • CAM (atmosphere)
  • POP (ocean)
  • CLM (land)

There are scripts in the each of the component shell_scripts directory which will build the filter program for that single component, copy the right files into the CESM caseroot directory, build CESM, and stage files to run an experiment where observations are only assimilated into that component.

The rest of this page describes setting up a coupled CESM run where you are assimilating into multiple active components.

Quickstart for CESM/DART on the NCAR 'yellowstone' machine

Both CESM and DART are large, complex software systems with a lot of power and capability, but they will require an investment of time and effort in learning how to set them up and run them correctly. The following list includes the tasks that need to be accomplished in order to run a CESM Data Assimilation (DA) experiment.

  • Get CESM
  • Get required patches to CESM1.1.1
  • Get DART
  • Initial conditions and observations
  • Running
  • Archiving files
  • Reset CESM files
  • Update DART files
  • Diagnostics

The following sections have some information that is specific to running CESM/DART on the NCAR 'yellowstone' machine. If you are running on another platform please contact us at dart@ucar.edu for help. The basics of the information below should still apply, but things like where the SourceMods are stored or where to find initial condition files will certainly differ.

Get CESM

Check out CESM version 1.1.1 (which we know is not the most recent release - but it is the latest version we have tested the DART code with). We are working on validating CESM 1.2 with DART and will update the code and documentation when it is ready.

Here's the user's guide for CESM version 1.1.1:

http://www.cesm.ucar.edu/models/cesm1.1/cesm/doc/usersguide/book1.html

Here's the page that explains how to download the release code:

http://www.cesm.ucar.edu/models/cesm1.1/cesm/doc/usersguide/x388.html

You will want "cesm1_1_1" as the model version. You will have to register as a user to check out the source code. There is a default username and password to access the subversion website.

This web page shows the names of the 'compsets' which are the configurations of the various models:

http://www.cesm.ucar.edu/models/cesm1.1/cesm/doc/modelnl/compsets.html

This page has pointers to the recent CESM versions:

http://www2.cesm.ucar.edu/models/current

If you want to try to build CESM separately from DART to learn about it, that's fine. But our scripts will need to rebuild it again with the DART configuration so you can wait on building it if you want.

Get required patches to CESM1.1.1

There are a few CESM files that we still need to modify to make things work with DART. In this dir on yellowstone is a tar file:

   ~nancy/cesm1_1_1_mods.tar
Copy that to your home dir and untar it. Our setup script will look for this subdirectory in your home directory and incorporate these patches when it builds CESM.

This file is also on the IMAGe web server, at this address: http://www.image.ucar.edu/pub/DART/CESM

Get DART

Go to the DART web pages: http://www.image.ucar.edu/DAReS and register as a DART user. (See the 'Download DART' item under the 'Getting Started' menu.) It's important for us to be able to document the size of our user base and everyone who registers helps us out.

Check out the "Lanai" release. The following instructions assume you've checked it out into a subdirectory "DART".

In the DART/mkmf directory copy the mkmf.template.intel.linux to mkmf.template . That should work for yellowstone with the default set of modules that are loaded on that machine. **IMPORTANT** If you are running with an active ocean, or you might ever sometime in the future run with an active ocean, you must add:

  -convert big_endian
to the list of FFLAGS in the mkmf.template file. POP uses some binary restart files that are still in the byte order for bluefire. If all your restart files for CAM, CAM-CHEM, CLM, etc are NetCDF files, then you can leave that flag off so yellowstone writes data in the normal little-endian byte order.

If you want to be able to reproduce runs exactly, you need to make sure this flag is part of the FFLAGS line:

  -fp-model precise
in your mkmf.template file. It has a slight performance penalty at runtime, but it ensures that floating point operations are executed in the same order inside the processor chip and the results should match bit for bit.

The DART scripts for CESM are in:

   DART/models/CESM/shell_scripts
Look at CESM1_1_1_setup_hybrid.csh.

There are some things you should customize at the top, including the compset, your project number for running, how many ensemble members, etc. Make any changes you want there.

Run the script. It will build the CAM, POP, and CLM filters first (the main DART executable is called filter), and then build and configure CESM. When the script is done you should have files in several directories. Unless you have changed these values in the setup script, the default locations are:

  • The 'cases' dir has the scripts that start and configure runtime vars for CESM. It also has the DART namelist files, input.nmls, for the various components. Usually we set this to 
    /glade/p/work/${USER}/cases
  • The actual execution directory is usually 
    /glade/scratch/${USER}/$casename/run
    and the executable files themselves are 
    /glade/scratch/${USER}/$casename/bld
  • When you've successfully run a model advance, the output files are saved in 
    /glade/scratch/${USER}/archive/$casename
    with a subdirectory for each component. Our output files will be in the 'dart' subdirectory, and under that directory will be 'hist', 'logs', and 'rest'. You'll find the DART diagnostic files and obs_seq.final files in the 'hist' subdirectory.

The CESM/DART system uses the CESM multi-instance capability to run multiple copies of components concurrently. The coupler routes data between the corresponding instances of each component. Each component can have N instances or 1 instance which is shared amongst all the multi-instances of other components. DART expects the CESM naming convension of adding instance numbers of the form _NNNN following the component name, e.g. my_case.cam_0001.r.2009-01-01-21600.nc is the first instance in a multi-instance case. For components without multiple instances the names follow the usual naming conventions, e.g. my_case.cam.r.2009-01-01-21600.nc.

Initial Conditions and Observations

ICs

DART requires an ensemble of initial conditions. This is the same as multiple instances in CESM terminology. If you have a multiple-instance run available to you for initial condition files, great. If you do not, one suggested option is to make N copies of a single initial condition file, each with the same data but with a different filename that includes the instance number. (See the paragraph immediately above discussing filename conventions.) When you run CESM for the first step it will be advancing N identical copies of the model. However, when you run DART with the following changes to the input.nml namelist, each instance will be perturbed differently and as the system runs these differences will evolve into distinct instances that will remain different.

Changes to the DART input.nml file to perturb identical initial conditions: 

&filter_nml :: start_from_restart = .false.,
&filter_nml :: restart_in_file_name = "filter_ics.0001",
&ensemble_manager_nml :: single_restart_file_in = .true.,
and if they aren't already set like this, 
&filter_nml :: inf_initial_from_restart    = .false.,    .false.,
&filter_nml :: inf_sd_initial_from_restart = .false.,    .false.,

After the first single job step, set the values back to the original settings: 

&filter_nml :: start_from_restart = .true.,
&filter_nml :: restart_in_file_name = "filter_ics",
&ensemble_manager_nml :: single_restart_file_in = .false.,
and if you are using prior inflation, 
&filter_nml :: inf_initial_from_restart    = .true.,    .false.,
&filter_nml :: inf_sd_initial_from_restart = .true.,    .false.,

Observations

One of the variables in the setup script specifies where the observation sequence files for the assimilation will be found. Because DART is running for only a single assimilation period at a time, the input obs_seq files for DART need to contain observations that are exactly within the assimilation window. For CAM, we usually run it in 6 hour advances, so the observation sequence files have to contain only 6 hours of observations. POP is often run for a day at a time between assimilations, so the POP observation files need to contain data for a full day. CLM has been run both in 6 hour assimilations and daily assimilations. In any case, the time period of the observations in each input file must match the length of time the model runs between assimilations.

The DART group has a collection of NCEP atmospheric obs, WOD (World Ocean Database) ocean obs, and some Land obs already converted and either on Glade disk on Yellowstone or archived on the HPSS. Contact the DART group if you are interested in what observations are already available.

Running

The setup script adds a call to the DART filter program inside the CESM run script, but initially it is commented out (on purpose). The intent is that you run the CESM case first, without DART, to ensure that the model is working correctly, has the right files, good namelist settings, etc. Then once you are sure CESM is working ok without DART, edit the case.run script. Look for the string "DART" and inspect the following script section. You should see a call to either the assimilate.csh script or perfect_model_obs.csh script, but it will be commented out with a leading # (hash) character. Remove the # and now you can either run the job again and start assimilating from the second job step, or you can reset the model to the initial time before you rerun. (See Resetting CESM files below for an easy script to do this.)

The DART run script will convert the CESM NetCDF files into the DART input format, run the assimilation program, and then convert the updated values back and overwrite the CESM files. If some error happens (out of disk space, bad node on the supercomputer, etc) during the execution of DART, you should use the script described in Resetting CESM files. This will reset the model data back to the start time for this job step and then you can rerun the run script.

with a subdirectory for each component. The DART output files will be in the 'dart' subdirectory, and under that directory will be 'hist', 'logs', and 'rest'. You'll find the DART diagnostic files and obs_seq.final files in the 'hist' subdirectory.

Resetting CESM files

The DART setup script creates several utility scripts which are customized for the particular case and location of files. They are put into the caseroot directory. One important script is called stage_cesm_files.

Running the stage_cesm_files script will restore the contents of the run directory to a previous step. If a job dies or an error occurs, this makes it easier to restart from some previous time.

To restart the experiment from the first job step, cd into the case directory. If it is not already false, run: 

./xmlchange CONTINUE_RUN false
and then run: 
./stage_cesm_files
The script will remove the current contents of the run directory and restage the files for the initial timestep.

To restart the experiment from a later job step, cd into the case directory. If it is not already true, run: 

./xmlchange CONTINUE_RUN true
Then edit the stage_cesm_files script and change the restart_time to be the day and time that you want to begin running from. e.g. if you want to restart at 0Z on Jan 2nd, 2001, you would set: 
set restart_time = 2001-01-02-00000
Then run: 
./stage_cesm_files
The script will remove the current contents of the run directory and restage the files for the specified time.

The script requires that the restart files are still on disk, either in the run directory or in the archive directory. If the long term archiver has moved the files onto tape and deleted them from disk, they will have to be restored before running this script.

Updating DART files

If you make any changes to the DART code and need to recompile the executables, use the standard DART quickbuild.csh scripts in the work directories. Once the executables are compiled, use the refresh_dart_files script in the caseroot directory to copy them to the correct location for this case.

Diagnostics

The DART release comes with a collection of Matlab® scripts for doing diagnostics both on the output observation sequence file (usually named obs_seq.final) and also on the state space values (usually found in Prior_Diag.nc and Posterior_Diag.nc). See the Diagnostics menu item on the DART web pages.

Miscellaneous and Futures

The current interface to the CESM climate modeling system does assimilation into CAM, POP, and CLM separately. Therefore at this point there is NOT a single CESM model_mod that is compiled into a single filter. However, there exists a beta version of a unified model_mod that redirects forward operator calls to the proper component-specific model_mod, and works around the conflicts between the naming of the model_mod at the module level. It would require a single state vector to fit into the memory space of one process, which means it would be limited in resolution. cesm_to_dart and dart_to_cesm converter prototypes also exist, which pack the data from each model end-to-end in the state vector.

NAMELIST

DART namelists are read from the file input.nml. Namelists start with an ampersand '&' and terminate with a slash '/'. Character strings that contain a '/' must be enclosed in quotes to prevent them from prematurely terminating the namelist.

Because each CESM component is assimilated separately, see the specific component documentation for namelist details:

FUTURE PLANS

Software development is ongoing that will allow observations to be assimilated into all CESM components at once, so low atmosphere observations can impact state values in both the ocean and land; surface ocean observations can impact the atmosphere. There are technical obstacles which are being removed to simply having this running; once it runs there will undoubtably be scientific questions that will have to be examined and answered.
